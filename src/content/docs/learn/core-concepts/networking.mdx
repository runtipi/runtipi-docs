# Networking & Reverse Proxy Explained

One of the most confusing aspects of self-hosting is networking. This guide breaks down the concepts in plain English so you can make informed decisions about exposing your apps.

## The Three Levels of Access

When you install an app with Runtipi, you need to decide who can access it. There are three levels:

### 1. 🏠 Local-Only Access (Default)

**Who can access?** Only you, on the same computer where Runtipi is running.

**URL Example:** `http://localhost:3000`

**Use case:** Testing an app before making it accessible to others.

**Security:** Most secure—nobody else can reach it.

---

### 2. 🏡 Local Network Access

**Who can access?** Anyone on your home/office network (same Wi-Fi/LAN).

**URL Example:** `http://192.168.1.100:3000` (your server's local IP)

**Use case:** Accessing apps from your phone, laptop, or other devices at home.

**Security:** Secure as long as you trust everyone on your network.

<Callout type="info">
  **This is the default for most Runtipi apps.** Access them via your server's IP address from any device on your network.
</Callout>

---

### 3. 🌍 Internet Access (Exposed)

**Who can access?** Anyone on the internet (if they know the URL).

**URL Example:** `https://cloud.yourdomain.com`

**Use case:** Accessing apps from anywhere (work, vacation, etc.) or sharing with friends.

**Security:** Requires careful setup—you're opening your server to the world.

## How Does Runtipi Handle Networking?

Runtipi uses a **reverse proxy** (Traefik) to manage all network traffic. Here's what that means:

### Without a Reverse Proxy (Traditional)

Each app would need its own port:
```
Nextcloud      → http://192.168.1.100:8080
File Browser   → http://192.168.1.100:8081
Jellyfin       → http://192.168.1.100:8096
...
```

**Problems:**
- Hard to remember ports
- Need to open many ports in firewall
- No automatic HTTPS
- Port conflicts

### With a Reverse Proxy (Runtipi's Way)

All traffic goes through port 80 (HTTP) or 443 (HTTPS):
```
http://nextcloud.local      → Traefik → Nextcloud container
http://files.local          → Traefik → File Browser container
http://jellyfin.local       → Traefik → Jellyfin container
```

**Benefits:**
- Easy-to-remember URLs
- Automatic HTTPS/SSL certificates
- One port to manage
- Apps don't conflict

<Callout emoji="💡">
  Think of Traefik as a receptionist at a building. All visitors come to one entrance (port 80/443), and the receptionist directs them to the right office (app container).
</Callout>

## Understanding Ports

A **port** is like an apartment number on your server. Each app needs its own number so traffic goes to the right place.

### Common Ports You'll See:
- **80**: HTTP (web traffic, not encrypted)
- **443**: HTTPS (web traffic, encrypted with SSL)
- **3000-9000**: Common app ports (Nextcloud, databases, etc.)

When Runtipi says an app is available on port `3001`, it means:
- **Internal**: The container uses port 3001
- **Access via**: Your reverse proxy (port 80/443) OR directly at `http://your-ip:3001`

<Callout type="warning">
  Most users should access apps through the Runtipi dashboard (which uses the reverse proxy), not by port numbers directly.
</Callout>

## What is SSL/HTTPS?

SSL (Secure Sockets Layer) encrypts the connection between your browser and the server. You want this for:

- **Security**: Data can't be intercepted (passwords, personal files, etc.)
- **Privacy**: Your ISP can't see what you're doing
- **Trust**: Browsers show a green padlock

### HTTP vs HTTPS

## Domain Names vs IP Addresses

### IP Address
- **Example:** `192.168.1.100` or `45.123.67.89`
- **Pros:** No setup needed, works immediately
- **Cons:** Hard to remember, changes sometimes, looks unprofessional

### Domain Name
- **Example:** `home.yourdomain.com` or `cloud.yourdomain.com`
- **Pros:** Easy to remember, looks professional, can have SSL certificates
- **Cons:** Costs money (usually $10-15/year), requires DNS configuration

<Callout>
  You can use Runtipi perfectly fine with just IP addresses on your local network. Domains are only necessary if you want custom URLs or internet access.
</Callout>

## Making the Decision: Should I Expose My App?

Use this decision tree:

```
Do you need to access this app from outside your home?
├─ NO → Keep it local network only (safest)
│        Access via: http://your-server-ip:port
│
└─ YES → Choose a method:
         ├─ Option 1: Cloudflare Tunnel (recommended)
         │            ✅ Most secure
         │            ✅ No port forwarding
         │            ✅ Free
         │            → Guide: /docs/guides/expose-apps-with-cloudflare-tunnels
         │
         ├─ Option 2: VPN (Tailscale/WireGuard)
         │            ✅ Very secure
         │            ✅ Works like local network
         │            ⚠️ Requires VPN client on devices
         │            → Research: Tailscale or WireGuard setup
         │
         └─ Option 3: Direct exposure (port forwarding)
                      ⚠️ Less secure
                      ⚠️ Exposes your IP
                      ✅ Easiest setup
                      → Guide: /docs/guides/expose-your-apps
```

## Common Scenarios

### Scenario 1: Family Media Server
**Need:** Access movies/music from any device at home

**Solution:** Local network access (default)
- No additional setup needed
- Access via `http://server-ip:port`
- Secure and fast

### Scenario 2: Personal Cloud Storage
**Need:** Access files from work, vacation, anywhere

**Solution:** Cloudflare Tunnel
- Secure, encrypted access from anywhere
- No exposed ports
- Free tier available
- Guide: [Expose with Cloudflare Tunnels](/docs/guides/expose-apps-with-cloudflare-tunnels)

### Scenario 3: Shared Photo Gallery
**Need:** Share photos with friends/family who don't live with you

**Solution:** Cloudflare Tunnel OR Direct Exposure with Strong Security
- Enable 2FA on the app
- Use strong passwords
- Regularly update apps
- Monitor access logs

## Security Best Practices

If you decide to expose apps to the internet:

1. ✅ **Use HTTPS** - Always encrypt connections
2. ✅ **Enable 2FA** - Two-factor authentication when available
3. ✅ **Strong passwords** - Use a password manager
4. ✅ **Keep updated** - Update Runtipi and apps regularly
5. ✅ **Limit exposure** - Only expose apps that truly need it
6. ✅ **Use Cloudflare Tunnel** - When possible, instead of port forwarding
7. ✅ **Monitor logs** - Check for suspicious access attempts
8. ✅ **Backup regularly** - In case something goes wrong

<Callout type="warning">
  **Never expose database management tools** (phpMyAdmin, pgAdmin, etc.) to the internet. These should always remain local-only.
</Callout>
